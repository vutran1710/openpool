# Matching Engine Design Spec

## Summary

Discovery engine for openpool. Profiles are organized into buckets by the indexer, encrypted using chain encryption, and explored client-side. Rate-limited by cryptographic work — no server gating needed.

## Architecture

Two components:

**Indexer** (server-side, GitHub Action):
- Has operator key — can decrypt all profiles
- Buckets profiles by operator-defined attribute partitions
- Builds chain-encrypted permutations per bucket
- Produces `index.db` — downloaded by clients

**Explorer** (client-side):
- Has local preferences — drives bucket selection
- Grinds chain encryption to unlock profiles one at a time
- Scores unlocked profiles against preferences
- Tracks seen profiles + learned constants for warm re-encounters

## Pool Configuration

Everything in `pool.yaml`:

```yaml
name: "<3 dating"
description: "terminal-native dating"
relay_url: wss://relay.example.com
operator_public_key: c251e2cf...

profile:
  age:
    type: range
    min: 18
    max: 100
  interests:
    type: multi
    values: hiking, coding, music, travel, food, photography, gaming, reading
  about:
    type: text
    required: false
  phone:
    type: text
    visibility: private
    required: false

roles:
  - man
  - woman

# --- Discovery configuration ---
indexing:
  partitions:
    - field: role
    - field: age
      step: 5
      overlap: 2
  permutations: 5
  difficulty: 20
```

### Indexing Config

| Field | Description |
|-------|-------------|
| `partitions` | List of attributes to bucket on. Each creates a dimension. |
| `partitions[].field` | Attribute name from `profile` or `roles` |
| `partitions[].step` | For range fields: bucket width (e.g., age step=5 → 18-22, 23-27...) |
| `partitions[].overlap` | For range fields: how many adjacent buckets a profile appears in |
| `permutations` | Number of shuffle orderings per bucket (default: 5) |
| `difficulty` | Single difficulty knob (default: 20). Controls cold unlock time. |

### Examples

```yaml
# Dating pool: role-first, then age ranges
indexing:
  partitions:
    - field: role
    - field: age
      step: 5
      overlap: 2
  permutations: 5
  difficulty: 20

# Job pool: role-first, then skills
indexing:
  partitions:
    - field: role
    - field: skills
  permutations: 3
  difficulty: 30
```

## Chain Encryption

### Key Components

Each encrypted profile entry has a key derived from three components:

| Component | Range | Source | Changes per permutation? |
|-----------|-------|--------|--------------------------|
| **hint** | 10^8 (8 digits) | Revealed by previous box in chain | Yes (different previous box) |
| **profile_constant** | 10^4 (4 digits) | `sha256(profile_content) % 10000` | No (fixed per profile) |
| **nonce** | `difficulty` | Random, generated by indexer | Yes (random per entry) |

### Key Derivation

```
ordering = pick_ordering(chain_context)   # one of 6 (3!) orderings
key = sha256("{a}:{b}:{c}")               # a,b,c = hint, constant, nonce in random order
ciphertext = AES-256-GCM(key, plaintext)
```

The ordering is deterministic per chain position but unknown to the explorer — adds 6x multiplier to search.

### Security Tiers

| Scenario | Search space | Time (at 10M attempts/s) |
|----------|-------------|--------------------------|
| **Without hint** | 10^8 × 10^4 × N × 6 | Days to years (infeasible) |
| **Cold** (have hint) | 10^4 × N × 6 | Seconds |
| **Warm** (know constant) | N × 6 | Instant |

### Chain Structure

Each bucket permutation is a chain of encrypted boxes:

```
Seed (public) → hint_0

Box 0: AES-GCM(key(hint_0, const_0, nonce_0), payload_0)
  payload_0 = { profile, my_constant: const_0, next_hint: hint_1 }

Box 1: AES-GCM(key(hint_1, const_1, nonce_1), payload_1)
  payload_1 = { profile, my_constant: const_1, next_hint: hint_2 }

...
```

- **Seed** is public (per-bucket, per-permutation). Provides `hint_0`.
- **next_hint** inside each box enables solving the next box.
- Without the hint, the search space is infeasible — enforces sequential access.
- **my_constant** is saved by the explorer for warm re-encounters in other permutations.

### Permutations

Each bucket has N permutations (pool-configurable, default 5). Same profiles, different shuffle order, different chain seeds.

```
Bucket [women, 25-30]:
  Permutation 0: Alice → Carol → Eve → Bob    (seed_0)
  Permutation 1: Bob → Eve → Alice → Carol    (seed_1)
  Permutation 2: Eve → Bob → Carol → Alice    (seed_2)
  ...
```

Explorer can jump between permutations. Known constants from any permutation apply everywhere — warm encounters are near-instant regardless of which permutation.

## Indexer

### Input
- All `.bin` files (decrypted with operator key)
- `pool.yaml` (partitions, difficulty, permutations)

### Process
1. Decrypt all profiles
2. Compute `profile_constant = sha256(profile) % 10000` per profile
3. Compute `hint = sha256("hint:" + profile) % 100000000` per profile
4. Assign profiles to buckets based on partition attributes
5. For each bucket:
   a. Generate N permutations (shuffled orderings)
   b. For each permutation: build chain-encrypted entries
6. Write `index.db`

### Output: `index.db`

```sql
-- Bucket metadata
CREATE TABLE buckets (
    bucket_id TEXT PRIMARY KEY,    -- e.g. "role:woman|age:25-30"
    partition_values TEXT,          -- JSON: {"role": "woman", "age_range": "25-30"}
    profile_count INTEGER
);

-- Permutation chains
CREATE TABLE chains (
    bucket_id TEXT,
    permutation INTEGER,
    seed BLOB,                     -- chain seed (provides first hint)
    UNIQUE(bucket_id, permutation)
);

-- Encrypted entries (ordered by position in chain)
CREATE TABLE entries (
    bucket_id TEXT,
    permutation INTEGER,
    position INTEGER,
    tag TEXT,                       -- match_hash (plaintext, for skip detection)
    ciphertext BLOB,               -- AES-256-GCM encrypted payload
    gcm_nonce BLOB,                -- 12-byte GCM nonce
    UNIQUE(bucket_id, permutation, position)
);
```

Uploaded as GitHub release asset (`index-latest` tag), same as current.

## Explorer

### Input
- `index.db` (downloaded from release asset)
- Local preferences (`~/.dating/pools/<pool>/preferences.yaml`)
- Seen history + known constants (`~/.dating/pool.db`)

### Preferences (local only, never committed)

```yaml
# ~/.dating/pools/test-pool/preferences.yaml
looking_for:
  role: woman
  age: { min: 25, max: 35 }
  interests: { overlap: 2 }
```

### Process
1. Load preferences
2. Select bucket based on preferences (e.g., `role:woman|age:25-30`)
3. Pick a permutation (random or round-robin)
4. Load checkpoint for this permutation (if resuming)
5. Iterate through chain entries:
   a. Read tag — if in seen set, advance chain key (instant)
   b. If known constant exists — solve warm (N × 6 attempts)
   c. Otherwise — solve cold (10^4 × N × 6 attempts)
   d. On success: save constant, update seen, score profile
6. Display scored profiles to user

### Scoring (per-unlock, informational)

Each profile is scored immediately upon unlock — displayed alongside the profile as a compatibility indicator. Scoring does NOT affect ordering — the chain order is fixed by the indexer. There is no ranking step.

| Primitive | Input | Output |
|-----------|-------|--------|
| `exact` | enum value | 1.0 if match, 0.0 if not |
| `proximity` | range value + tolerance | Linear decay: 1.0 at diff=0, 0.0 at diff=tolerance |
| `overlap` | multi values + min count | 1.0 if shared >= count, scales down below |

Score is purely informational — a reference for the user when deciding to like or skip. The chain encryption controls discovery rate; the bucket selection controls relevance; scoring is just a visual aid.

### Local Storage

```sql
-- In ~/.dating/pool.db

-- Known constants (carry across permutations)
CREATE TABLE constants (
    match_hash TEXT PRIMARY KEY,
    profile_constant INTEGER,
    learned_at INTEGER
);

-- Explorer checkpoints (resume position per permutation)
CREATE TABLE checkpoints (
    bucket_id TEXT,
    permutation INTEGER,
    position INTEGER,           -- last solved position
    chain_hint INTEGER,         -- hint for next box (to resume)
    UNIQUE(bucket_id, permutation)
);

-- Seen profiles
CREATE TABLE seen (
    match_hash TEXT PRIMARY KEY,
    seen_at INTEGER,
    action TEXT                  -- skip, like, pass
);
```

## Matching Modes

### Pair Matching (Dating)

- Roles are labels (same fields for all)
- Partitions by role: user browses the opposite role's bucket
- Mutual interest required (same issue-based flow as today)
- Scoring: proximity on age, overlap on interests

### Role-Based Matching (Job)

- Roles have different fields (`employer.requirements` vs `candidate.skills`)
- Partitions by role + key attributes (skills, experience level)
- Mutual interest still required (employer likes candidate AND candidate likes employer)
- Scoring: overlap on skills, proximity on experience

The system doesn't distinguish between these modes in code. The difference is entirely in `pool.yaml` configuration — partition fields, role-specific attributes, and local preferences.

## Data Flow

```
Operator writes pool.yaml
  ↓
Indexer (Action cron):
  decrypt .bin files → bucket → chain-encrypt → index.db → release asset
  ↓
Client downloads index.db
  ↓
Explorer:
  preferences (local) → select bucket → grind chain → unlock profiles → score → display
  ↓
User swipes:
  like → create interest issue (existing flow)
  skip → mark seen locally
```

## Migration from Current System

| Current | New |
|---------|-----|
| `index.db` with flat profiles table | `index.db` with buckets + chains + entries |
| Profiles visible in plaintext in index | Profiles chain-encrypted |
| No rate limiting on discovery | Chain encryption rate-limits unlocking |
| Vector encoding + bilateral filters | Post-unlock scoring with preferences |
| `FilterValues` + cosine similarity | Simple scoring primitives (exact, proximity, overlap) |
| `map[string]bool` seen | `seen` table with timestamps |
| No preference storage | `preferences.yaml` local file |
| Server-side matching logic | All matching client-side |
